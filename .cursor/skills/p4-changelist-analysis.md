# P4 变更列表影响分析

分析一个 Perforce 变更列表，输出：修改内容摘要、受影响模块映射、下游影响和优先级测试计划。

## 工作流

```
输入（CL 号）
  │
  ├─ 步骤 1：收集 ──► p4 describe / p4 diff
  ├─ 步骤 2：分类 ──► 将文件映射到游戏模块
  ├─ 步骤 3：追溯 ───► 搜索引用以确定下游影响
  └─ 步骤 4：输出 ──► 摘要 + 测试计划
```

### 步骤 1 — 收集变更详情

在项目终端（PowerShell）中运行：

```powershell
# 基本信息：作者、描述、文件列表及操作类型
p4 describe -s <CL>

# 统一差异用于代码级分析
p4 describe -du <CL>
```

如果 CL 是**搁置**（pending）状态，使用：

```powershell
p4 describe -S -du <CL>
```

从输出中解析：
- **作者**和**描述**
- **文件列表**：仓库路径、操作类型（add/edit/delete/integrate）、文件类型
- **差异块**：每个文件中哪些行发生了变化

P4 命令行详情见 [p4-reference.md](p4-reference.md)。

### 步骤 2 — 分类到游戏模块

使用 [module-map.md](module-map.md) 中的项目路径约定，将每个变更文件路径映射到游戏模块。

构建表格：

| 文件路径（工作区相对路径） | 操作 | 模块 | 子系统 |
|-------------------------------|--------|--------|------------|
| `Assets/Beyond/Scripts/Gameplay/System/MissionSystem/...` | edit | 任务 | 任务链 |
| `Assets/Beyond/DynamicAssets/Gameplay/Dialog/Json/...` | edit | 对话 | 对话树 |
| ... | ... | ... | ... |

### 步骤 3 — 追溯下游影响

对每个修改的文件，在工作区中搜索引用：

```powershell
# C# 引用
在 *.cs 中搜索类名/方法名

# Lua 引用
在 *.lua 中搜索模块名/函数名

# JSON 交叉引用（如 scriptId, missionId, dialogId）
在其他 JSON/CS/Lua 文件中搜索变更 JSON 中出现的关键 ID
```

特别注意：
- **配置 ID 引用**：变更的 `scriptId`、`missionId`、`questId`、`npcProxyId`、`interactiveId` 可能被很多其他文件引用
- **序列化类型变更**：修改 `[Serializable]` 类或 `[JsonProperty]` 字段可能导致所有使用该类型的资产反序列化失败
- **Lua 模块依赖**：修改 Lua 文件的返回值或公共 API 会影响所有 `require()` 调用者
- **Prefab/UI 绑定**：修改绑定到 UI Prefab 节点的 C# 字段名可能静默破坏面板

## 核心原则

1. **始终先执行 p4 命令** — 不要猜测文件内容，要读取实际的差异
2. **按 ID 追溯，而非按路径** — 任务 ID 或脚本 ID 可能在远离定义它的文件处被引用
3. **风险 = 影响范围 x 破坏概率** — 共享基类中的一行拼写修复比新增一个独立文件风险更高
4. **测试场景必须可复现** — 包含地图名、任务 ID、任务步骤、预期行为
5. **不确定模块负责人时要说明** — 标注"负责人不明"比猜错更好
